SEQUENCE OF OPERATIONS
----------------------
1. Customer sends a text inquiry (items, quantities, delivery deadline, request date).
2. Orchestrator Agent receives the request and extracts dates/deadlines.
3. Inventory Agent:
     a. Maps informal item names to exact catalog names
     b. Checks stock via get_stock_level / get_all_inventory
     c. Places supplier restock orders via create_transaction when stock is low
     d. Returns parsed items JSON for downstream agents
4. Quoting Agent:
     a. Searches historical quotes via search_quote_history
     b. Calculates line-item pricing with tiered bulk discounts
     c. Returns a customer-friendly quote explanation
5. Sales Agent:
     a. Verifies fulfillment feasibility against current stock
     b. Estimates delivery using get_supplier_delivery_date
     c. Finalizes the sale via create_transaction (sales type)
     d. Reports updated financial status via generate_financial_report
6. Orchestrator assembles all agent outputs into one customer response.

AGENT SUMMARY (4 agents + orchestrator = 5 maximum)
----------------------------------------------------
| Agent          | Responsibility                                      |
|----------------|-----------------------------------------------------|
| Orchestrator   | Route requests, coordinate agent sequence           |
| Inventory      | Stock queries, reorder decisions, supplier orders   |
| Quoting        | Historical pricing, bulk discounts, quote generation  |
| Sales          | Order fulfillment, delivery timelines, transactions |

STEP 2 — STARTER CODE FUNCTION REVIEW
================================================================================

Each helper function in project_starter.py and its role in the system:

init_database(db_engine, seed=137)
  Creates SQLite tables (transactions, quote_requests, quotes, inventory),
  loads CSV data, seeds initial cash ($50,000) and stock order transactions.
  Called once at the start of run_test_scenarios().

generate_sample_inventory(paper_supplies, coverage=0.4, seed=137)
  Randomly selects 40% of catalog items and assigns stock quantities (200–800)
  and minimum reorder thresholds (50–150). Used internally by init_database.

create_transaction(item_name, transaction_type, quantity, price, date)
  Inserts a row into the transactions table. Types: 'stock_orders' (purchases
  that reduce cash) or 'sales' (revenue that reduces inventory). Used by
  place_restock_order and finalize_sale tools.

get_all_inventory(as_of_date)
  Returns a dict of {item_name: stock} for all items with positive stock as of
  a given date, computed from net transactions. Used by get_inventory_snapshot.

get_stock_level(item_name, as_of_date)
  Returns current stock for one item as of a date. Used by check_items_availability,
  auto_restock_if_needed, and finalize_sale.

get_supplier_delivery_date(input_date_str, quantity)
  Estimates supplier delivery lead time: same-day (≤10), 1 day (≤100),
  4 days (≤1000), 7 days (>1000). Used by place_restock_order,
  estimate_delivery_date, and auto_restock_if_needed.

get_cash_balance(as_of_date)
  Computes net cash (total sales revenue minus total stock purchase costs).
  Used internally by place_restock_order to verify reorder affordability.

generate_financial_report(as_of_date)
  Produces cash balance, inventory valuation, total assets, itemized inventory
  summary, and top-selling products. Used by get_financial_status tool and
  run_test_scenarios() for progress tracking.

search_quote_history(search_terms, limit=5)
  Joins quotes with quote_requests and searches by keyword in request text
  and quote explanations. Used by search_historical_quotes tool.

================================================================================
STEP 3 — FRAMEWORK SELECTION: smolagents
================================================================================

Chosen framework: smolagents (Hugging Face)

Rationale:
  - Used throughout the Udacity course demos and exercises
  - ToolCallingAgent pattern maps cleanly to worker-agent specialization
  - @tool decorator integrates Python functions as agent tools with JSON schema
  - OpenAIServerModel supports Vocareum proxy (api_base configuration)
  - Lightweight orchestration without heavy infrastructure

Configuration:
  model_id  = "gpt-4o-mini"
  api_base  = "https://openai.vocareum.com/v1"
  api_key   = UDACITY_OPENAI_API_KEY (from .env)


================================================================================
STEP 4 — IMPLEMENTATION DETAILS
================================================================================

File: project_starter.py (single submission file)

OrchestratorAgent.handle_request() runs a 3-phase pipeline:
  Phase 1 → InventoryAgent  (stock check + restock)
  Phase 2 → QuotingAgent    (historical lookup + pricing)
  Phase 3 → SalesAgent      (fulfillment + transaction recording)

Key design decisions:
  - Item alias resolution (_resolve_item_name) bridges informal customer
    language ("A4 glossy paper") to exact DB names ("Glossy paper")
  - Tiered bulk discounts encourage larger orders (5%–15% by volume)
  - auto_restock_if_needed triggers before quoting so large orders can be
    accommodated when cash allows
  - Sales agent checks delivery feasibility before recording transactions
  - Request dates are always passed to tools to maintain temporal accuracy

STRENGTHS
---------
1. Modular agent design — each agent has focused tools and clear responsibilities
2. Database-backed decisions — stock, cash, and quotes come from real SQLite queries
3. Bulk discount strategy — tiered pricing encourages larger orders and matches
   historical quote patterns in quotes.csv
4. Automatic restocking — inventory agent proactively orders from suppliers when
   stock is insufficient, using get_supplier_delivery_date for lead times
5. Item alias mapping — handles common customer phrasing without DB errors
6. Temporal accuracy — all queries use the request date, not today's date

AREAS FOR IMPROVEMENT
---------------------
1. Item name parsing relies on LLM + alias table; items not in catalog (e.g.,
   "A3 paper", "balloons", "tickets") cannot be fulfilled — a dedicated
   parsing agent or NER model would improve extraction accuracy
2. Sequential agent calls (3 LLM round-trips per request) are slow and costly;
   a single orchestrator with all tools could reduce latency and API budget
3. Restock orders are recorded immediately but supplier delivery may miss the
   customer's deadline — the system should more strictly block finalize_sale
   when get_supplier_delivery_date exceeds the requested delivery date
4. Quote totals are parsed from free-text agent output; structured JSON output
   from the quoting agent would eliminate parsing errors
5. No persistent conversation memory — repeat customers are treated independently

SUGGESTED FUTURE IMPROVEMENTS
------------------------------
1. Add structured output (Pydantic models) for parsed items and quote totals
2. Implement deadline-aware fulfillment gate that compares supplier delivery
   date vs. customer deadline before allowing finalize_sale
3. Add a lightweight regex/LLM pre-processor to extract line items before
   agent delegation, reducing tool-call errors
4. Cache historical quote searches for similar event types (ceremony, parade)
5. Add inventory forecasting based on upcoming request patterns in quote_requests.csv
6. Parallelize independent tool calls within each agent phase

ARCHITECTURE DECISION-MAKING (why this design?)
------------------------------------------------
1. Chose sequential orchestration over parallel because each phase depends on
   the prior: inventory status informs quoting, quotes inform sales decisions.
2. Separated Quoting from Sales so pricing logic (discounts, history) does not
   mix with transaction recording — single responsibility per agent.
3. Put reorder logic in Inventory Agent (not Sales) because restocking is a
   supply-chain decision that must happen before fulfillment is attempted.
4. Used smolagents ToolCallingAgent because course materials demonstrate this
   pattern and it maps cleanly to tool → helper function wrappers.
5. Wrapped all 7 starter helpers as tools rather than calling them directly from
   agents, ensuring auditable database interactions and rubric compliance.
